OOMKilled — Memory Limit Too Low restart_deployment → resolved

Pod vẫn tồn tại (khác Scenario A) nhưng bị kernel kill vì vượt memory limit → OOMKilled → CrashLoopBackOff. Agent nhận ra đây là transient crash, không phải config error → chọn restart_deployment (không phải scale_deployment). Sau đó restore limit → pod lên bình thường.

Tại sao test này quan trọng: Scenario A dễ vì "không có pod nào" là signal hiển nhiên. Scenario A' yêu cầu agent phân biệt OOMKilled (restart được) vs config error (không restart được — Scenario B). Đây là reasoning thật.
  1. Inject — set memory limit cực thấp 
    kubectl set resources deployment/ride-service -n vroom-dev \
  --limits=memory=6Mi --requests=memory=6Mi
# ride-service idle dùng ~9Mi → 6Mi limit đảm bảo OOMKilled ngay khi start — đợi ~20s
kubectl get pods -n vroom-dev -l app=ride-service
# Expected: STATUS=OOMKilled hoặc CrashLoopBackOff, RESTARTS≥1

# Confirm lý do kill
kubectl describe pod -n vroom-dev -l app=ride-service | grep -A3 "Last State"
# Expected: Reason: OOMKilled
  2. Open n8n — Test mode → fire webhook 
    curl -s -X POST http://192.168.242.10:30078/webhook-test/vroom-alert \
  -H "Content-Type: application/json" \
  -d '{
    "alerts": [{
      "labels": {
        "alertname": "PodCrashLooping",
        "service":   "ride-service",
        "namespace": "vroom-dev",
        "severity":  "critical"
      },
      "status": "firing"
    }]
  }'
  3. Verify agent output trong n8n + Slack
    FieldExpectedNếu sai
    Remediation toolrestart_deploymentSolver đang chọn none → xem log agent
    ConfidenceHIGH hoặc MEDIUM—
    Suggested commandkubectl rollout restart deployment/ride-service -n vroom-dev—
    Slack button✅ APPROVE + ❌ REJECT (cả hai)Nếu chỉ có ACKNOWLEDGE → remediation_tool=none, xem bước dưới
    kubectl logs -n monitoring deployment/incident-agent --tail=30 \
  | grep -E "^\[(rewoo|remediate)"
# [rewoo:solver] raw=...restart_deployment...
# [remediate] alert=PodCrashLooping service=ride-service approved=True tool=restart_deployment
    Expected evidence in Slack after this implementation:
    • [get_events]: Warning OOMKilling pod/ride-service-xxx Container ride-service was using Xmi, above the limit of 6Mi (Warning type → kept; Normal lifecycle events dropped)
    • [get_logs]: Previous container logs (RESTARTS≥1 triggers --previous automatically); if the same exit message repeated, collapsed to [×N] signal: killed
    • [get_metrics]: No running pods — live metrics unavailable — pod is in CrashLoopBackOff; metrics-server has no data during crash window
  4. Restore memory limit TRƯỚC khi click APPROVE

    Nếu restart chạy mà limit vẫn còn 6Mi, pod sẽ OOMKilled ngay lập tức và outcome = failed. Restore trước — cần restore cả init containers vì inject step (no -c) đã set tất cả containers xuống 6Mi:

  -c copy-migrations --limits=memory=32Mi --requests=memory=16Mi
kubectl set resources deployment/ride-service -n vroom-dev \
  -c migrate --limits=memory=64Mi --requests=memory=32Mi

# Main container
kubectl set resources deployment/ride-service -n vroom-dev \
  -c ride-service --limits=cpu=200m,memory=128Mi --requests=cpu=50m,memory=32Mi

# Verify main container updated
kubectl get deployment/ride-service -n vroom-dev \
  -o jsonpath='{.spec.template.spec.containers[0].resources}' | python3 -m json.tool
# Expected: "limits": {"cpu": "200m", "memory": "128Mi"}
  5. Click ✅ APPROVE trong Slack → xác nhận remediation

    Browser mở approve URL → Wait node resume → Switch output 0 → Call Incident Agent — Remediate → kubectl rollout restart chạy → agent đợi 35s → check pod health → Slack nhận outcome message.

    # Verify pod restarted thành công
kubectl get pods -n vroom-dev -l app=ride-service
# Expected: READY=1/1  STATUS=Running  RESTARTS tăng thêm 1

kubectl logs -n monitoring deployment/incident-agent --tail=10 \
  | grep remediate
# [remediate] waiting 35s for pod restart...
# [remediate] post-restart: 1/1 Running → outcome=resolved

kubectl set resources deployment/ride-service -n vroom-dev \

CrashLoop — Bad REDIS_ADDR none + dev hint → acknowledge

REDIS_ADDR=bad-host:6379 → startup DNS failure → CrashLoopBackOff → agent diagnoses config error → outputs none with HIGH confidence + dev hint — no automated fix possible.

  1. Inject the failure 
    kubectl set env deployment/ride-service -n vroom-dev REDIS_ADDR=bad-host:6379
# ride-service exits at startup — wait ~30 s for CrashLoopBackOff
kubectl get pods -n vroom-dev -l app=ride-service
# Expected: STATUS=CrashLoopBackOff, RESTARTS≥1
  2. Open n8n — switch webhook to Test mode

    Same as Scenario A — click Listen for test event on the Webhook node.

  3. Fire the alert webhook 
    curl -s -X POST http://192.168.242.10:30078/webhook-test/vroom-alert \
  -H "Content-Type: application/json" \
  -d '{
    "alerts": [{
      "labels": {
        "alertname": "PodCrashLooping",
        "service":   "ride-service",
        "namespace": "vroom-dev",
        "severity":  "critical"
      },
      "status": "firing"
    }]
  }'
  4. Watch n8n + agent logs
    n8n NodeWhat to verify
    HTTP → /investigateremediation=null, confidence=HIGH
    Build Approvaldev_hint block present — shows REDIS_ADDR fix
    Human ApprovalAction reads Acknowledge (no auto-fix available)
    kubectl logs -n monitoring deployment/incident-agent -f \
  | grep -E "^\[(rewoo|memory|remediate)"
# [rewoo:solver] raw=...none..."confidence":"HIGH"...
# [rewoo:hint] Dev action: REDIS_ADDR env var points to...
# [remediate] no approved remediation → outcome=skipped
    Expected: confidence=HIGH, remediation_tool=none. The 3rd LLM call generates a dev hint pointing to the bad env var. Solver should NOT output restart_deployment — a DNS lookup failure is a config error.
    Expected evidence in Slack after this implementation:
    • [get_events]: Warning BackOff pod/ride-service-xxx Back-off restarting failed container ride-service
    • [get_logs]: Previous container logs (RESTARTS≥1 → --previous); DNS error repeated on every startup → collapsed: [×N] dial tcp: lookup bad-host …: no such host; successful health probe lines (/healthz | 200 |) are stripped
    • No [get_metrics] in evidence — metrics may be absent or very low (pod crashes before metrics-server collects data; planner may skip the tool)
  5. Acknowledge in n8n → Slack shows dev hint

    Click Acknowledge. Slack message shows root cause, dev hint, and no suggested kubectl command (outcome=skipped).

  6. Restore the cluster 
    # Trailing '-' removes the env var
kubectl set env deployment/ride-service -n vroom-dev REDIS_ADDR-
kubectl rollout status deployment/ride-service -n vroom-dev
kubectl get pods -n vroom-dev -l app=ride-service
# Expected: STATUS=Running

Scenario A: Switch luôn route sang REJECT thay vì Call Incident Agent — Remediate

Switch đọc field decision từ "Extract Decision and Execution ID" node. Kiểm tra node đó có đang map đúng không:

Sau khi Wait resume, click vào node Extract Decision and Execution ID trong n8n canvas → xem Output. Field decision phải là "approved". Nếu là undefined hoặc null, node chưa đọc đúng query param từ approve URL.

Fix: mở node "Extract Decision and Execution ID", đảm bảo nó map:

decision      ← {{ $json.query.decision }}
execution_id  ← {{ $json.query.execution_id }}

Kiểm tra Switch condition đang dùng gì:

# Output 0 (→ Remediate) phải match khi:
{{ $json.decision }} equals "approved"

# Output 1 (→ REJECT) là fallback

Nếu approve URL không reachable từ browser (browser không vào được cluster), Wait node không bao giờ resume → timeout → đi REJECT. Kiểm tra WEBHOOK_URL trong n8n deployment:

kubectl get deployment n8n -n monitoring -o jsonpath='{.spec.template.spec.containers[0].env}' \
  | python3 -m json.tool | grep -A2 WEBHOOK_URL
# Phải là http://192.168.25.139:30078/

# Fix nếu sai:
kubectl set env deployment/n8n -n monitoring WEBHOOK_URL=http://192.168.25.139:30078/
groq-secret missing after cluster reprovision 
kubectl get sealedsecret groq-secret -n monitoring
# If SealedSecret exists but Secret doesn't — controller issue:
kubectl logs -n kube-system -l app.kubernetes.io/name=sealed-secrets --tail=20
